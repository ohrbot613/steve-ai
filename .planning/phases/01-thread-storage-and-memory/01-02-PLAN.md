<objective>
Create the ThreadMemoryService that handles all thread lifecycle operations: creating threads, saving messages, loading conversation context with token-aware windowing, and summarizing older messages.

Purpose: This service is the bridge between raw MongoDB data and the LangChain agent. It takes a threadId and returns ready-to-use LangChain message objects with intelligent context management. The agent (Phase 2) will call this service before every invocation to get conversation history. Without this, threads are just inert data.

Output: A single service file in `utils/` that exports ThreadMemoryService with methods for the complete thread memory lifecycle.
</objective>

  <name>Task 1: Create ThreadMemoryService with thread and message CRUD</name>
  <files>PDF automation/utils/memoryService.js</files>
  <action>
Create `PDF automation/utils/memoryService.js` using CommonJS (matching codebase convention).

Create a `ThreadMemoryService` class with these methods:

**Constructor:**
```javascript
constructor(options = {})
```
- `maxTokens`: default 4000 (tokens reserved for conversation history)
- `summaryModel`: default `'openai/gpt-4o-mini'` (cheaper model for summarization, using OpenRouter model naming to match existing codebase pattern in LangChainController.js)
- `defaultMessageLimit`: default 20 (initial message fetch limit before token trimming)
- Store the OpenRouter config for LLM instantiation (apiKey from `process.env.OPEN_ROUTER`, baseURL `https://openrouter.ai/api/v1`, with standard headers). Match the exact ChatOpenAI config pattern from LangChainController.js.

**createThread(userId):**
- Generate UUID v4 using `const { v4: uuidv4 } = require('uuid')`
- Create and return a Thread document with `{ threadId: uuidv4(), userId }`
- Return the created thread document (caller needs `thread.threadId`)

**getThread(threadId, userId):**
- Query `Thread.findOne({ threadId, userId })` -- ALWAYS validate ownership
- If not found, throw an error: `'Thread not found or access denied'`
- Return the thread document

**saveMessage(threadId, role, content, metadata = {}):**
- Create and return a Message document with `{ threadId, role, content, metadata }`
- Do NOT validate thread ownership here (caller is responsible -- this keeps the method fast for the hot path where ownership was already checked)

**getMessages(threadId, options = {}):**
- `limit`: default `this.defaultMessageLimit` (how many recent messages to fetch from DB)
- Query `Message.find({ threadId }).sort({ createdAt: 1 }).lean()`
- If limit is provided, use a strategy: fetch the last N messages by first counting, then using skip. Or simpler: fetch all sorted ascending and take the last N. Given these are per-thread (not millions), fetching all and slicing is fine.
- Return raw message documents (not LangChain objects -- that's loadContext's job)

**loadContext(threadId, userId):**
This is the key method that Phase 2 will call. It must:

1. Call `this.getThread(threadId, userId)` to validate ownership (throws if invalid)
2. Fetch ALL messages for the thread: `Message.find({ threadId }).sort({ createdAt: 1 }).lean()`
3. If no messages, return `{ messages: [], summary: null, totalMessages: 0, includedMessages: 0 }`
4. Convert to LangChain message objects:
   - `role === 'user'` -> `new HumanMessage(content)`
   - `role === 'assistant'` -> `new AIMessage(content)`
   - `role === 'system'` -> `new SystemMessage(content)`
5. Use LangChain's `trimMessages` for token-aware windowing:
   ```javascript
   const { trimMessages } = require("@langchain/core/messages");
   ```
   Create a ChatOpenAI instance for token counting (use the same OpenRouter config as the rest of the app). Call:
   ```javascript
   const trimmed = await trimMessages(lcMessages, {
       maxTokens: this.maxTokens,
       strategy: 'last',
       tokenCounter: llm,
       allowPartial: false
   });
   ```
6. If `trimmed.length < lcMessages.length`, call `this._summarizeOlder()` on the messages that were cut (the first `lcMessages.length - trimmed.length` messages)
7. Return:
   ```javascript
   {
       messages: summary ? [summary, ...trimmed] : trimmed,
       totalMessages: lcMessages.length,
       includedMessages: trimmed.length,
       summarized: summary !== null
   }
   ```

**_summarizeOlder(messages):** (private helper)
- Takes an array of LangChain message objects (the ones that were trimmed out)
- Builds a conversation text from them: `${msg._getType()}: ${msg.content}`
- Creates a summarization prompt that is SPECIFIC to this business domain:
  ```
  Summarize this conversation history concisely.
  CRITICAL: Preserve all specific details including:
  - Invoice numbers, amounts, dates
  - Vendor/supplier names
  - File names or document references
  - Unresolved questions or pending actions
  - Key decisions made
  - Any tool results or data lookups performed

  Keep the summary under 200 words but do NOT omit specific identifiers.

  Conversation:
  {conversationText}
  ```
- Call the summary LLM (gpt-4o-mini via OpenRouter, temperature 0) to generate summary
- Return `new SystemMessage('[Previous conversation summary: ${summary.content}]')`

**Important implementation notes:**
- Use `require` not `import` (CommonJS project)
- The LLM config MUST match LangChainController.js pattern: `apiKey: process.env.OPEN_ROUTER`, `configuration: { baseURL: "https://openrouter.ai/api/v1", defaultHeaders: { ... } }`
- Do NOT use deprecated LangChain memory classes (ConversationBufferWindowMemory, etc.)
- Do NOT use LangChain's MongoDBChatMessageHistory (has known bugs per research)
- Export: `module.exports = ThreadMemoryService;` (export the class, not an instance -- callers instantiate with their own options)
