Create `PDF automation/modals/threadModal.js` following the existing codebase conventions (CommonJS, same style as userModal.js and processModal.js).

The Thread schema must have:
- `threadId`: String, required, unique, indexed. This is the UUID v4 identifier (generated externally via `uuid` package, NOT auto-generated by Mongoose).
- `userId`: ObjectId, ref "User", required, indexed. References the authenticated user who owns the thread.
- Schema options: `{ timestamps: true }` to auto-add `createdAt` and `updatedAt` (this is the codebase convention, see processModal.js).

Add these indexes:
- `threadId: 1` (already via `unique: true` on field, but be explicit)
- `userId: 1` (for "list threads by user" queries)
- Compound index `{ threadId: 1, userId: 1 }` (for ownership validation queries: "find thread X owned by user Y")

Export as `mongoose.model("Thread", threadSchema)`.

Follow the exact style of processModal.js: require mongoose, define schema, add indexes after schema definition, create model, module.exports.

Do NOT add any methods, statics, or virtuals. Keep it minimal -- just the schema and indexes.

<task type="auto">
  <name>Task 2: Create Message Mongoose model</name>
  <files>PDF automation/modals/messageModal.js</files>
  <action>
Create `PDF automation/modals/messageModal.js` following the same codebase conventions.

The Message schema must have:
- `threadId`: String, required, indexed. Links message to its thread (matches Thread.threadId). NOT an ObjectId -- it's a string UUID for consistency with how threadId is passed around.
- `role`: String, required, enum `['user', 'assistant', 'system']`. The 'system' role is needed for summary messages.
- `content`: String, required. The message text.
- `metadata`: Object (Mixed type), optional, default `{}`. Stores tool call info, usage stats, etc. Sub-fields:
  - `toolCalls`: Array of Mixed (captures LangChain tool call objects)
  - `usage`: Mixed (captures token usage if available)
- Schema options: `{ timestamps: true }` for automatic createdAt/updatedAt.

Add these indexes:
- `threadId: 1` (for all messages in a thread)
- Compound index `{ threadId: 1, createdAt: 1 }` (for chronological retrieval of messages within a thread -- this is the primary query pattern)

Export as `mongoose.model("Message", messageSchema)`.

Follow the exact style of existing modals. Keep it minimal.
  </action>
  <verify>
Run: `node -e "const m = require('./PDF automation/modals/messageModal'); console.log(m.modelName, Object.keys(m.schema.paths));"` from the project root. Should output "Message" and include threadId, role, content, metadata, createdAt, updatedAt, _id, __v.

Also verify indexes: `node -e "const m = require('./PDF automation/modals/messageModal'); console.log(JSON.stringify(m.schema.indexes()));"` should show the compound index {threadId:1, createdAt:1}.

Verify role enum: `node -e "const m = require('./PDF automation/modals/messageModal'); console.log(m.schema.path('role').enumValues);"` should output ['user', 'assistant', 'system'].
  </verify>
