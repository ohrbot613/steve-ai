    **Step 1: Install better-sse**
    Run `cd "PDF automation" && npm install better-sse` to add the SSE library.

    **Step 2: Create the streaming service at `PDF automation/services/streamingService.js`**

    This is a NEW file (the `services/` directory does not exist yet -- create it).
    Use CommonJS module format (`require`/`module.exports`) to match the codebase convention.

    The service exports a single function: `streamAgentResponse(req, res, options)`.

    Options parameter: `{ agent (llmWithTools), input (string), threadId (string|null), userId (string), systemMessages (array of LangChain messages), context (object), conversationHistory (array) }`

    Implementation details:

    1. **Create SSE session** using `better-sse`'s `createSession(req, res)`. This automatically sets Content-Type, Cache-Control, Connection headers and manages keep-alive.

    2. **Token accumulation**: Initialize `let fullResponse = ''` for buffer-then-save.

    3. **AbortController**: Create an AbortController for cancellation on client disconnect.
       ```javascript
       const abortController = new AbortController();
       req.on('close', () => {
         abortController.abort();
       });
       ```

    4. **Stream from LangChain**: Use the existing `llmWithTools.stream(messages)` pattern (same as the existing streaming code in LangChainController.js lines 349-358). Iterate with `for await (const chunk of streamResponse)`:
       - If `chunk.content` exists: accumulate to `fullResponse`, send as SSE event via `session.push({ token: chunk.content }, 'token')`
       - If `chunk.tool_calls` exists and has entries: send `session.push({ tool: chunk.tool_calls[0].name }, 'tool_start')` -- this lets the frontend show "Using tool: X" status
       - Check `abortController.signal.aborted` at the start of each iteration and break if true

    5. **After stream completes** (for-await loop finishes):
       - Import Message model: `const Message = require('../modals/messageModal')`
       - Import Thread model: `const Thread = require('../modals/threadModal')`
       - Import uuid: `const { v4: uuidv4 } = require('uuid')`
       - If no threadId was provided, create a new thread:
         ```javascript
         const newThreadId = uuidv4();
         await Thread.create({ threadId: newThreadId, userId });
         threadId = newThreadId;
         ```
       - Save both user and assistant messages to MongoDB:
         ```javascript
         await Message.insertMany([
           { threadId, role: 'user', content: input },
           { threadId, role: 'assistant', content: fullResponse, metadata: { toolCalls: toolCallsUsed } }
         ]);
         ```
       - Send completion event: `session.push({ done: true, threadId }, 'end')`

    6. **Error handling**: Wrap in try/catch. On error, send `session.push({ error: error.message }, 'error')`. In finally block, clean up resources.

    7. **Do NOT call `res.end()`** -- better-sse manages the connection lifecycle. The session will end when the stream finishes or the client disconnects.

    IMPORTANT: The existing streaming code in LangChainController (lines 287-369) uses `llmWithTools.stream(messages)` which returns chunks with `.content`. This is different from `agent.streamEvents()`. Use the SAME pattern as the existing code since the LLM+tools binding is already proven. Do NOT use `streamEvents()` -- it's for LangGraph agents, and this codebase uses `llm.bindTools()` with manual tool execution loops.

    For this first implementation, stream the initial LLM response tokens. The full tool-calling loop (iterative tool execution) will remain non-streaming for now -- only the final AI response generation should stream. The approach:
    - The service should accept pre-built messages array and the bound LLM
    - Stream the LLM response
    - If tool_calls come back instead of content, fall back to non-streaming tool execution loop, then stream the final response

    This matches the existing architecture where tool execution is iterative and only the final response needs streaming.
